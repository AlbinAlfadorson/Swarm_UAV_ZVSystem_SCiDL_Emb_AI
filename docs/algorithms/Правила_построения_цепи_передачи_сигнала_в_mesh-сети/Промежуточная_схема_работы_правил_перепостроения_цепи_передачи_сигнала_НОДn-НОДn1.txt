Общее, промежуточное описание обязательных примеров для первого стенда {Proteus, KiCAD, Wireshark, Gazebo + ROS2 + NS-3, Ubuntu\Fedora}

Видимые ноды: N2, N3, N4 для N1
если (N3 - связь потеряна ❌)
• Запрос траэктории до NQ для возможности передать данные на прямую, в случае успеха передача на прямую от N1 к NQ, в противном случае ->
• Поиск ближайшего Nn от N1 из числа {N2, N4..}
• N2: ближе к NQ? - ДА, сила сигнала: 0.8 ✓
• N4: ближе к NQ? - НЕТ (дальше), соответственно сила сигнала, к примеру: 0.3 - ❌
Выбираем N2 (единственный подходящий) ->
У N2 есть заведомо готовый путь: N2 → N4 → N5 → Королева
→ N1 отправляет остальным N: "Изменение топологии: N1 сменил маршрут с N3 на N2"
→ N1 отправляет на NQ информацию о перестроении маршрута для себя с целью сбижения с N2 для дальнейшей передачи  
→ данных по этой цепи (в случае необходимости) : "Изменение топологии: N1 сменил маршрут с N3 на N2"
→ N1 продолжает движение по своей траектории, либо если маршрут был перестроен для передачи данных он возвращается на свою старую позицию в случае если преоритет данных выше чем у всех остальных в новой цепи передачи и тогда либа вся цепь перестраивается под приоритет найденного N1 события, либа остаётся на своих старых позициях (взависимости от предполётных деректив), если не, то - даёт сигнал на королеву через новую цепь, что возвращается обратно либа, двигается по новой траэктории если события у прочих участников цепи выше!
